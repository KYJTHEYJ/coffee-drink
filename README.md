# 설계 의도
- 다중 서버, 다중 DB 를 지닌 환경에서 구동 되는 것을 가정하여 다양한 방법을 구성
  - 각 Entity 별 ID를 UUID v7 적용, JWT, Redisson 분산락 적용, Kafka 처리를 통한 이벤트 처리
  - Flyway migration 을 통한 다중 DB 환경 관리 지원
  - 주문 실패시 보상 트랜잭션을 통하여 데이터의 정합성 유지

# 문제 해결 전략과 해결 과정
1. 포인트 충전, 사용시 로그 생성을 통한 정합성 기반 마련
  - 같은 회원의 경우 충전과 사용시 로그 이벤트가 다른 파티션으로 분산되면 로그 생성 순서가 바뀔 수 있음
    - **key를 MemberID 로 정하여 같은 파티션에 해당 될 수 있도록 마련**
   
2. 주문시 Kafka 이벤트들의 발행 시점 관련
  - 직접 메세지를 발행하게 하면 주문 트랜잭션이 실패해도 메세지가 발행이 되어버림
    - **이를 방지하기 위해 EventPublisher 를 사용하여 커밋 완료 시점에 Kafka 이벤트 메세지들이 발행되도록 처리**
   
3. AOP Self-Invocation 사용 방지 및 중첩 트랜잭션 방지 해결
  - 주문시 재고와 포인트 사용에 있어 분산락을 사용하여 동시성 제어를 한 상황
  - 분산 락은 REQUIRES_NEW 를 통하여 별도의 트랜잭션으로 락 해제 전 비즈니스 로직이 커밋되도록 보장하게 하였음
  - 주문 정보를 저장하는 과정에 트랜잭션 어노테이션을 적용하면 이중 커넥션이 적용됨, Connection Pool 고갈
    - 단순히 트랜잭션 어노테이션을 없애도 문제는 해결되지만, OrderService 의 책임이 다소 커지게 됨
      - (주문 저장, 포인트 차감, 재고 조정 등의 모든 책임을 지님)
    - **트랜잭션 어노테이션을 없애고 독립적으로 작동시키면서 책임을 줄이는 방법은 저장과 로직 흐름을 따로 가져가게 하면 되므로 Facade 패턴을 적용하여 흐름 제어와 저장을 분리**
   
4. DB 분산 환경 가정시 각 테이블 식별자 결정 과정
  - 단순히 AUTO_INCREMENT 로 1..2..3... 계속 증가하게 하면 다중 환경 DB에서 키 중복이 날 가능성은 매우 높음
  - **ULID, TSID 등 여러가지 방법을 찾았지만 표준에 가장 걸맞는 것은 UUID라 생각되어 시간 기반 정렬이 되어있는 UUID v7 으로 결정**

5. DB 분산 환경 가정시 각 테이블 설정 문제
  - 따로 마이그레이션을 위한 코드를 수동으로 지니게 하지 않는 한 DB 환경은 각각 틀어질 가능성이 매우 높음
  - **FlyWay 라이브러리를 이용하여 마이그레이션 코드를 지니게 하면 모든 DB에서 환경을 동일하게 가져갈 수 있음**
